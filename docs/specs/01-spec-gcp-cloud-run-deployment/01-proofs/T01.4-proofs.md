## Design notes

- **SAs**: `create_sa` uses describe-then-conditional-create; the two
  SAs are created only on a fresh project, skipped on re-runs.
- **Bindings**: use `add-iam-policy-binding`, which is idempotent by
  gcloud design (re-adding the same member+role leaves the policy
  document unchanged). No describe-first guard needed.
- **Scope minimization**: `storage.objectAdmin` is bucket-scoped (not
  project-wide) and `iam.serviceAccountUser` is SA-scoped (not
  project-wide). These are the two bindings where scope choice
  materially affects the security posture.
- **`--condition=None`** is explicit on project-level bindings — avoids
  the "add unconditionally / ignore conditional bindings" interactive
  prompt that gcloud otherwise raises when the existing policy has any
  conditional bindings.
- **`run.admin` retained**: spec flags `run.developer` as a possible
  tightening. Deferred to a follow-up per the spec's guidance to verify
  empirically before narrowing.
