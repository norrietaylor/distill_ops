# T02.1 Proof Artifacts — Cloud Run Service Definition

## Summary

Task T02.1 implements Unit 2 (Cloud Run Service Definition) of the GCP Cloud Run deployment spec. This unit delivers the Docker image, Distillery configuration, and Cloud Run service manifest needed to run Distillery on Cloud Run with durable DuckDB storage on GCS FUSE.

**Status: COMPLETE**

## Files Created

### 1. `gcp/Dockerfile`
- **Purpose:** Extends the public `ghcr.io/norrietaylor/distillery:${IMAGE_TAG}` base image with Cloud Run-specific configuration
- **Key Features:**
  - Sets `FASTMCP_HOME=/data/fastmcp` for OAuth state persistence across revisions
  - Copies and configures `gcp/distillery-gcp.yaml` as the application config
  - Sets `DISTILLERY_CONFIG=/app/distillery-gcp.yaml`
  - Fixes `/data` ownership at runtime (handles FUSE mount ownership)
  - Drops privileges to `appuser` before launching the application
  - Exposes port 8000
- **Verification:** Syntax and structure validated; full build test deferred to integration environment

### 2. `gcp/distillery-gcp.yaml`
- **Purpose:** Application configuration for Cloud Run deployment with DuckDB and GitHub OAuth
- **Key Features:**
  - `storage.backend: duckdb` with `database_path: /data/distillery.db` (GCS FUSE mount)
  - `embedding.provider: jina` with API key from Secret Manager (`JINA_API_KEY` env var)
  - `server.auth.provider: github` with client credentials from Secret Manager
  - `server.webhooks.enabled: true` with secret from Secret Manager
  - Rate limiting configured (500 daily embedding budget, 900 MB DB size limit)
- **Verification:** Valid YAML; parity with existing `fly/distillery-fly.yaml` confirmed; comments document required secrets

### 3. `gcp/service.yaml`
- **Purpose:** Knative serving Cloud Run service manifest defining the complete deployment
- **Key Features:**
  - **Execution environment:** `gen2` (required for GCS FUSE support)
  - **Service account:** `distillery-run` (Workload Identity provided by bootstrap)
  - **Resource limits:** 2 vCPU, 1024 MB memory (meets >= 1024 MB requirement)
  - **Health probes:** Startup and liveness probes on `/.well-known/oauth-authorization-server`
  - **Container concurrency:** 10 (matches Fly hard_limit)
  - **Request timeout:** 300 seconds (accommodates embedding-heavy polls)
  - **GCS FUSE volume:** Mounted at `/data`, backed by `{PROJECT_ID}-distillery-data` bucket
  - **Secret Manager integration:** 5 secrets referenced (JINA_API_KEY, GITHUB_CLIENT_ID, GITHUB_CLIENT_SECRET, DISTILLERY_WEBHOOK_SECRET, DISTILLERY_BASE_URL)
  - **Traffic:** Latest revision receives 100% traffic

### Design Considerations

- ✓ Dockerfile mirrors Fly Dockerfile structure and patterns
- ✓ Configuration follows existing distillery YAML schema (matching fly/distillery-fly.yaml)
- ✓ Service manifest follows Knative serving conventions
- ✓ All sensitive values use Secret Manager references (no plaintext in manifest)
- ✓ Health probes target OAuth endpoint (validates application readiness)
- ✓ GCS FUSE volume writable (`readOnly: false`) to support DuckDB writes
- ✓ Startup probe grace period (initialDelaySeconds=5, periodSeconds=5, failureThreshold=6) provides ~30s tolerance for cold start + FUSE mount

- **Unit 3 (gcp-deploy.yml)** will follow: CI/CD workflow to build and deploy container image

## Deployment Notes

- Service deployment requires: `gcloud run services replace gcp/service.yaml --region=us-central1 --project=<PROJECT_ID>`
- Before deployment, operator must create 5 Secret Manager secrets with production values
- Image path in service.yaml will be substituted by CI workflow: `{REGION}-docker.pkg.dev/{PROJECT_ID}/distillery/distillery:{TAG}`
- Service will scale from 0 to 1 instance based on traffic (min_instances=0, max_instances=1)
