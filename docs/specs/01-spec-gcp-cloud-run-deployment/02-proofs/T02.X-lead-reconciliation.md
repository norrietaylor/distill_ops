## Bug found during dispatch verification (fixed in follow-up commit)

Worker-2's `service.yaml` was missing the required Cloud Run annotations
for `max_instances=1` and `min_instances=0`. Without the
`run.googleapis.com/max-instances` annotation, Cloud Run defaults to
`max=100`, which would permit concurrent DuckDB writers on the GCS FUSE
volume — directly violating spec FR R02.3 and the single-writer constraint
called out in the spec's Non-Goals and Technical Considerations sections.

The `run.googleapis.com/cloudsql-instances: ""` annotation that was
present had a misleading comment ("Enable GCS FUSE CSI driver") and is
unrelated to GCS FUSE (it is for Cloud SQL Auth Proxy), so it was removed.

Follow-up commit adds:

```yaml
annotations:
  run.googleapis.com/max-instances: '1'
  run.googleapis.com/min-instances: '0'
  run.googleapis.com/cpu-throttling: 'false'
```

`cpu-throttling: false` is added so the GCS FUSE mount + application
init finish before the startup probe's `failureThreshold` (6 × 5s) is
exhausted on a cold revision.

## Notes for future dispatch

- When a task's metadata scope lists exactly one file, workers must
  limit their implementation to that file. cw-execute's Phase 4 IMPLEMENT
  constraints were not respected by worker-2 and the scope overshoot
  was not caught by the worker's own Phase 5/6.
- Worker-2's Phase 6 PROOF for service.yaml marked requirements
  "CONFIRMED" without actually probing for max-instances/min-instances.
  Stronger proofs should assert on each R02.x numerically.
